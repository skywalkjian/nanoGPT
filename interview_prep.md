日期：2026-03-29

目标：把 `nanoGPT` 从“学过一个项目”升级成“能支持 AI research lab 面试表达的项目”。

## 核心原则

对于 AI research lab 面试，光说“我跑通了 nanoGPT”不够。

你需要让面试官看到这 4 件事：

1. 你能快速建立系统级理解，而不是只会跟 README 走。
2. 你能解释核心训练与建模机制，而不是只会背 Transformer 术语。
3. 你能提出有研究味道的问题和实验，而不是只会做使用者视角总结。
4. 你能诚实描述局限、失败案例和下一步计划。

一句话说：

> 面试里真正有价值的，不是“我把项目跑起来了”，而是“我能把它拆开、分析、验证、提出改进问题”。

## 面试目标重定义

你今天不需要把自己包装成“大模型专家”。

你真正要达成的是：

1. 能在 1 分钟内讲清楚这个仓库是什么。
2. 能在 3 到 5 分钟内讲清楚训练链路和模型链路。
3. 能回答 10 个左右的技术追问。
4. 能讲出至少 3 个你自己提出的实验或改进方向。
5. 能区分“这个项目做到了什么”和“这个项目没做到什么”。

## 你必须掌握的 10 个问题

### 1. 这个项目的完整执行路径是什么

标准答案骨架：

```text
原始文本
-> 预处理得到 train.bin / val.bin / meta.pkl
-> train.py 用 memmap 取 batch
-> model.py 进行前向传播得到 logits 和 loss
-> 训练循环更新参数并周期性保存 checkpoint
-> sample.py 加载 checkpoint 并生成文本
```

### 2. `x` 和 `y` 分别是什么

标准答案骨架：

```text
x 是输入 token 序列，
y 是向右错开一位的目标序列。
训练目标是根据前面的 token 预测下一个 token。
```

### 3. 为什么这是自回归语言模型

标准答案骨架：

```text
因为模型在每个位置只能使用左侧上下文来预测下一个 token，
而不能访问未来 token。
这正是 causal mask 的作用。
```

### 4. `CausalSelfAttention` 在做什么

标准答案骨架：

```text
先把输入投影成 q、k、v，
再计算每个位置对历史位置的注意力权重，
最后加权汇总 value。
causal mask 保证每个位置不能看未来。
```

### 5. 为什么 `Block` 是 attention + MLP + residual

标准答案骨架：

```text
attention 负责跨 token 交互，
MLP 负责位置内的非线性变换，
residual 让深层训练更稳定，也保留原始信息路径。
```

### 6. 为什么训练时和推理时 forward 不一样

标准答案骨架：

```text
训练时需要所有位置的 logits 来计算整体 loss，
推理时只需要最后一个位置的 logits 来采样下一个 token，
所以 sample 时只取最后位置更高效。
```

### 7. `gradient_accumulation_steps` 在解决什么问题

标准答案骨架：

```text
它用多个 micro-batch 累积梯度，来模拟更大的有效 batch size，
这样在显存有限时仍然能维持较稳定的训练设置。
```

### 8. 为什么 `get_batch()` 用 memmap

标准答案骨架：

```text
因为训练数据是大的一维 token 流，
memmap 可以避免一次性把整个数据集读进内存，
并支持快速随机切片。
```

### 9. checkpoint 里保存了什么，为什么重要

标准答案骨架：

```text
保存了模型权重、优化器状态、模型配置、当前迭代数、最佳验证损失和训练配置。
这样既能恢复训练，也能保证推理和复现实验时参数一致。
```

### 10. 这个仓库和真正研究/工业训练系统差在哪

标准答案骨架：

```text
它是最小实现，不是完整平台。
缺少更完整的数据管道、实验管理、评测体系、容错、多种并行策略、
更复杂的 tokenizer/packing、监控、超参搜索和规模化训练工程能力。
但正因为它足够小，才适合作为理解和快速实验的起点。
```

## 你可以提出的研究问题

下面这些问题不需要你今天全做完，但你最好能讲出来。

### 问题 1：tokenization 粒度对建模行为有什么影响

你可以说：

```text
字符级 tokenization 更简单，但序列更长，局部模式学习更直接；
BPE tokenization 序列更短，更贴近实际 LLM 训练，但预处理和 vocabulary 更复杂。
我会把它当成一个关于建模单位与上下文利用效率的对比问题。
```

### 问题 2：`block_size`、模型大小和训练稳定性之间有什么关系

你可以说：

```text
更大的 block_size 提供更长上下文，但会增加计算和显存成本；
更大的模型容量可能在小数据上更容易过拟合。
我会做一个小型 ablation，看验证损失、收敛速度和生成质量怎么变化。
```

### 问题 3：dropout 在小数据 finetune 和预训练场景里的角色是否不同

你可以说：

```text
README 里也提示预训练常用更低 dropout，而小数据 finetune 可能更需要正则。
我会把它作为一个数据规模与正则策略匹配的问题去看。
```

### 问题 4：最小实现里的哪些设计是“为了可读性”，哪些是“为了性能”

你可以说：

```text
例如 model.py 单文件很利于理解，
而 fused AdamW、Flash Attention、compile 更偏性能。
这个区分很重要，因为研究原型和生产系统对代码结构的诉求不一样。
```

## 如果面试官追问“你最大的收获是什么”

建议答法：

```text
我最大的收获不是 Transformer 这个名词本身，
```

## 如果面试官追问“这个项目有什么局限”

这是加分题，不要回避。

你可以说：

1. 它是教学和研究原型友好的最小实现，不是完整训练平台。
2. 它没有覆盖现代 LLM 系统的全部关键部分，比如更复杂的数据打包、评测、并行策略和实验管理。
3. 它对理解机制非常好，但不代表你已经理解大规模系统的全部工程挑战。
